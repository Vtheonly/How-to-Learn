# MOC — Protocols

> Eight step-by-step playbooks. Each one is a sequenced recipe, not a list of tips.

Protocols exist because methods are general — "retrieve, space, interleave" — but tasks are specific. A protocol translates the methods into the **exact sequence of steps** you should follow for a specific task type.

## How protocols are structured

Every protocol follows the same shape:

1. **When to use** — trigger conditions.
2. **Inputs** — what you need before starting.
3. **Outputs** — what you should have when done.
4. **Steps** — numbered, sequenced, with time boxes.
5. **Common traps** — where people go wrong.
6. **Linked methods** — which study methods each step operationalizes.
7. **Linked theory** — which cognitive principle each step exploits.

## The prototyping habit

When you face a task that doesn't fit any protocol, **write a new one**. Start from the closest existing protocol, adapt the steps, and tag it `#protocol-draft`. After three uses, promote it to `04_Protocols/` with a real number.

The vault is a living system. The protocols you write yourself will be more useful than the eight provided here, because they are tuned to your actual tasks.
