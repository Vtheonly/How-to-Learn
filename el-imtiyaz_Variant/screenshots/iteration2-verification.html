Issue §1 / #1 FATAL

Broken inter-rule data flow (TOTAL CREANCE always = 0)

BeforeLedgerService.computeFields() evaluated each rule against the SAME stale ctx.fields. The TOTAL CREANCE rule ("devisAnnuel - totalVersements") always returned 0 because neither field was written back to ctx after being computed. EVERY row showed totalCreance = 0 regardless of payments.
AfterAfter each rule evaluation, the result is written back to ctx.fields. The pipeline is now a real DAG: J → L (devisAnnuel) ─┐ R–Y → P (totalVersements) ─┴→ Q (totalCreance) TOTAL CREANCE now computes correctly for every row.
computeFields() writes devisAnnuel, totalVersements, totalCreance back to ctx

Issue §17 / #17 MEDIUM

Formula context missing optionCode/level/classCode/destination

BeforebuildFormulaContext() only injected numeric fields. Formula rules could not reference optionCode, level, classCode, or destination — even though these strings are exactly what the Excel operator uses to decide which formula components to include. A rule like IF(optionCode = "TRNSP", reg + tuition + transport, reg + tuition) was impossible.
Afterctx.fields now includes optionCode, level, classCode, destination, and hasTransport (boolean). User-defined rules can branch on any of these. The starter DEVIS ANNUEL rule's watchedFields was extended to include level, optionCode, destination so the rule re-evaluates when these change.
